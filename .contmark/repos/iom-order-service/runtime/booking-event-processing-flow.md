1. `BookingConsumer.listenForBookingMessage` receives a `Booking` Avro record, builds `EventContextProvider` with topic/offset metadata, and delegates to `EventDispatcher.dispatch(...)`. (source: src/main/kotlin/com/maersk/iom/messaging/BookingConsumer.kt:26-42)
2. `EventDispatcher` selects the first `BookingEventHandler` whose `supports(eventType)` matches; `null` routes to `BookingUpdatesEventHandler`, while `PRICING_UPDATE` routes to `PricingUpdatesEventHandler`. (source: src/main/kotlin/com/maersk/iom/messaging/EventDispatcher.kt:12-19, src/main/kotlin/com/maersk/iom/messaging/BookingUpdatesEventHandler.kt:46-48, src/main/kotlin/com/maersk/iom/messaging/PricingUpdatesEventHandler.kt:36-38)
3. Booking-status flow validates allowed booking/work-process payloads, loads the persisted plan by service-plan number and acceptable stage, rejects stale work-process sequences, optionally enriches missing Telikos config, changes `servicePlanStage`, and merges booking payload details. (source: src/main/kotlin/com/maersk/iom/messaging/BookingUpdatesEventHandler.kt:49-66, src/main/kotlin/com/maersk/iom/model/Extensions.kt:30-63)
4. The handler persists through `orderService.save(..., isBookingStatusChanged = true)` and may call `executeBooking` or `confirmBooking` when single-click feature flags and work-process milestones allow it. (source: src/main/kotlin/com/maersk/iom/messaging/BookingUpdatesEventHandler.kt:73-91, src/main/kotlin/com/maersk/iom/messaging/BookingUpdatesEventHandler.kt:101-138)
5. Pricing-update flow loads the order, extracts `AUTOGENERATED` booking and equipment charges, rewrites them to `MANUAL` retrieval mode with IOM charge refs, and saves the updated plan. (source: src/main/kotlin/com/maersk/iom/messaging/PricingUpdatesEventHandler.kt:38-52, src/main/kotlin/com/maersk/iom/messaging/PricingUpdatesEventHandler.kt:66-124)
6. `OrderServiceImpl.save()` enriches the aggregate, adds planning deadline/customer-order handling, persists with `repository.save(...)`, then emits `SERVICE_PLAN_CHANGED` plus optional status/booking-status/repricing domain events. (source: src/main/kotlin/com/maersk/iom/order/domain/OrderServiceImpl.kt:180-225, src/main/kotlin/com/maersk/iom/order/domain/OrderServiceImpl.kt:323-349)
7. `DomainEventHandlerFacade` resolves typed handlers from Spring and `OutboxStrategy` persists each event to `outbox_messages` with priority and `PENDING` status instead of publishing directly in this repo. (source: src/main/kotlin/com/maersk/iom/model/event/handler/DomainEventHandlerFacade.kt:13-38, src/main/kotlin/com/maersk/iom/model/event/strategy/OutboxStrategy.kt:13-35)
8. Both booking handlers currently terminate the reactive chain with blocking waits (`toFuture().get()` or `.block()`), and acknowledgments are optional based on config flags. (source: src/main/kotlin/com/maersk/iom/messaging/BookingUpdatesEventHandler.kt:87-91, src/main/kotlin/com/maersk/iom/messaging/PricingUpdatesEventHandler.kt:47-52)
